## 두 도구의 코드 작성 사이클

Claude Code로 코드를 작성할 때는 **Plan Mode → 계획 검토 → ExitPlanMode → acceptEdits** 라는 명확한 4단계 사이클을 따랐습니다. 계획 단계에서 파일을 건드리지 않고 의도를 먼저 확인할 수 있었습니다.

Codex는 현재 **Plan Mode → 작업 실행 → sandbox/approval 보호장치** 흐름을 제공합니다. 공식 Best practices 문서는 복잡하거나 모호한 작업에서는 먼저 계획을 요청하고, `/plan` 또는 `Shift`+`Tab`으로 Plan Mode를 켤 수 있다고 설명합니다. sandbox와 approval은 계획 자체를 대체한다기보다, 구현 단계에서 파일·명령 실행 권한을 제어하는 안전장치입니다.

## sandbox_mode와 approval_policy 조합

Codex의 안전성은 두 설정의 조합으로 결정됩니다.

| `sandbox_mode` | `approval_policy` | 동작 |
|---------------|-------------------|------|
| `read-only` | `on-request` | 읽기만 가능, 쓰기 시 승인 요청 |
| `workspace-write` | `on-request` | 쓰기 가능, 각 도구 실행 전 승인 |
| `workspace-write` | `never` | 완전 자동 실행 (CI 환경 등) |
| `danger-full-access` | `never` | 모든 제한 해제 (`--yolo` 와 동일) |

```toml
# ~/.codex/config.toml
sandbox_mode = "workspace-write"
approval_policy = "on-request"
```

또는 명령행에서 직접 지정할 수 있습니다.

```bash
codex -s workspace-write -a on-request "테스트 파일을 추가해 줘"
```

## Plan-first 패턴

Claude Code의 Plan Mode에 익숙하다면 Codex에서도 "계획 먼저, 구현 나중" 흐름을 그대로 가져가면 됩니다. 차이는 구현 단계에서 sandbox와 approval 정책을 함께 설계한다는 점입니다.

**패턴 1: Plan Mode로 먼저 구체화**

```bash
# TUI에서 /plan 또는 Shift+Tab 사용
codex

# 예시 프롬프트
/plan
인증 모듈의 세션 만료 버그를 고치고 싶다.
먼저 관련 파일을 읽고, 재현 방법과 최소 수정 계획을 제안해 줘.
아직 파일은 수정하지 마.
```

**패턴 2: read-only로 먼저 분석**

```bash
# 1단계: 읽기 전용으로 코드 분석
codex -s read-only "인증 모듈 구조를 분석하고 개선 방향을 제안해 줘"

# 2단계: 내용 확인 후 실제 수정
codex resume --last -s workspace-write "방금 제안한 대로 수정해 줘"
```

**패턴 3: approval_policy로 단계별 승인**

```bash
# 구현 중 각 승인 지점을 직접 확인
codex -s workspace-write -a on-request "리팩터링을 진행해 줘"
```

:::tip[팁]
`/plan`은 대화형 세션에서 가장 자연스럽고, `read-only`는 CLI 실행 자체를 읽기 전용으로 묶고 싶을 때 유용합니다. 두 방법은 배타적이지 않으므로, 위험한 리팩터링에서는 `read-only` 세션에서 계획을 만든 뒤 `workspace-write`로 이어받는 방식을 사용할 수 있습니다.
:::

## diff 승인 워크플로

Codex에서 `approval_policy = "on-request"` 를 사용하면 파일 수정 전에 diff를 확인하고 승인 또는 거부할 수 있습니다. Desktop App에서는 이 과정이 시각적 diff 뷰어로 제공됩니다.

```bash
# TUI에서 승인 흐름
# [y] 승인  [n] 거부  [e] 설명 요청  [a] 이후 모두 승인
```

:::caution[주의]
`--dangerously-bypass-approvals-and-sandbox` (또는 `--yolo`) 플래그는 모든 승인과 sandbox를 우회합니다. 로컬 개발 중 빠른 반복에는 편리하지만, 프로덕션 코드나 CI 환경에서는 사용을 피하세요.
:::

<ConceptCard title="granular approval_policy">

`approval_policy = { granular = { ... } }` 형식을 사용하면 승인 항목별로 더 세밀한 정책을 적용할 수 있습니다. Codex Rules(developers.openai.com/codex/rules)와 함께 사용하면 특정 명령에만 더 높은 권한을 부여하고 자동 실행할 수 있습니다.

```toml
approval_policy = { granular = { sandbox_approval = true, rules = true } }
```

</ConceptCard>

## 핵심 정리

- Codex는 `/plan` 또는 `Shift`+`Tab`으로 Plan Mode를 사용할 수 있으며, sandbox와 approval은 구현 단계의 안전장치입니다
- **`workspace-write` + `on-request`** 조합이 Claude Code의 `acceptEdits` 모드와 유사한 검토 지점을 제공합니다
- **`/plan` → 계획 검토 → 구현 → `/review`** 흐름을 기본 루프로 삼으면 변경 범위와 검증 기준을 함께 관리할 수 있습니다
- **`codex -s read-only` → 분석 → `codex resume --last -s workspace-write`** 패턴은 파일 변경을 물리적으로 막고 싶을 때 유용합니다
- `approval_policy = "never"` 는 CI/비대화형 환경에 적합합니다
- `--yolo` 플래그는 모든 제한을 해제하므로 신중하게 사용해야 합니다
