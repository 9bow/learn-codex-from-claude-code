## 두 도구의 코드 작성 사이클

Claude Code로 코드를 작성할 때는 **Plan Mode → 계획 검토 → ExitPlanMode → acceptEdits** 라는 명확한 4단계 사이클을 따랐습니다. 계획 단계에서 파일을 건드리지 않고 의도를 먼저 확인할 수 있었습니다.

Codex는 이 사이클을 **sandbox 설정 → 작업 실행 → approval 정책** 이라는 방식으로 대체합니다. Plan Mode라는 개념 자체가 없지만, sandbox와 approval 조합으로 유사한 안전성을 확보할 수 있습니다.

<Compare>
  <Fragment slot="left">
  **Claude Code**
  - `/plan` 입력 → Plan Mode 진입
  - 모델이 수정 계획만 서술 (파일 미변경)
  - 계획 확인 후 `ExitPlanMode` 호출
  - 파일 변경 시 `acceptEdits` 권한 필요
  - 권한 모드: `plan` / `acceptEdits` / `auto`

  </Fragment>
  <Fragment slot="right">
  **Codex**
  - Plan Mode 없음
  - `sandbox_mode = "read-only"` 로 읽기 전용 실행
  - 작업 확인 후 `sandbox_mode = "workspace-write"` 로 변경
  - `approval_policy = "on-request"` 로 각 작업 승인
  - `approval_policy = "never"` 로 자동 실행

## sandbox_mode와 approval_policy 조합

Codex의 안전성은 두 설정의 조합으로 결정됩니다.

| `sandbox_mode` | `approval_policy` | 동작 |
|---------------|-------------------|------|
| `read-only` | `on-request` | 읽기만 가능, 쓰기 시 승인 요청 |
| `workspace-write` | `on-request` | 쓰기 가능, 각 도구 실행 전 승인 |
| `workspace-write` | `never` | 완전 자동 실행 (CI 환경 등) |
| `danger-full-access` | `never` | 모든 제한 해제 (`--yolo` 와 동일) |

```toml
# ~/.codex/config.toml
sandbox_mode = "workspace-write"
approval_policy = "on-request"
```

또는 명령행에서 직접 지정할 수 있습니다.

```bash
codex -s workspace-write -a on-request "테스트 파일을 추가해 줘"
```

## Plan Mode 우회 패턴

Claude Code의 Plan Mode처럼 "파일을 건드리지 않고 먼저 분석하게 하고 싶다"는 요구는 Codex에서 다음 패턴으로 대응합니다.

**패턴 1: read-only로 먼저 분석**

```bash
# 1단계: 읽기 전용으로 코드 분석
codex -s read-only "인증 모듈 구조를 분석하고 개선 방향을 제안해 줘"

# 2단계: 내용 확인 후 실제 수정
codex resume --last -s workspace-write "방금 제안한 대로 수정해 줘"
```

**패턴 2: approval_policy로 단계별 승인**

```bash
# 각 도구 실행 전 승인 요청 → 계획과 동일한 제어감
codex -s workspace-write -a on-request "리팩터링을 진행해 줘"
```

:::tip[팁]
`codex resume --last` 는 마지막 세션을 이어받습니다. read-only 분석 결과를 그대로 컨텍스트로 유지한 채 workspace-write 세션을 시작할 수 있습니다.
:::

## diff 승인 워크플로

Codex에서 `approval_policy = "on-request"` 를 사용하면 파일 수정 전에 diff를 확인하고 승인 또는 거부할 수 있습니다. Desktop App에서는 이 과정이 시각적 diff 뷰어로 제공됩니다.

```bash
# TUI에서 승인 흐름
# [y] 승인  [n] 거부  [e] 설명 요청  [a] 이후 모두 승인
```

:::caution[주의]
`--dangerously-bypass-approvals-and-sandbox` (또는 `--yolo`) 플래그는 모든 승인과 sandbox를 우회합니다. 로컬 개발 중 빠른 반복에는 편리하지만, 프로덕션 코드나 CI 환경에서는 사용을 피하세요.
:::

<ConceptCard title="granular approval_policy">

`approval_policy = "granular"` 를 사용하면 도구 유형별로 다른 승인 정책을 적용할 수 있습니다. Codex Rules(developers.openai.com/codex/rules)와 함께 사용하면 특정 명령에만 더 높은 권한을 부여하고 자동 실행할 수 있습니다.

```toml
approval_policy = "granular"
```

</ConceptCard>

## 핵심 정리

- Codex에는 **Plan Mode가 없습니다.** `sandbox_mode = "read-only"` 가 가장 가까운 대안입니다
- **`workspace-write` + `on-request`** 조합이 Claude Code의 `acceptEdits` 모드와 유사한 안전성을 제공합니다
- **`codex -s read-only` → 분석 → `codex resume --last -s workspace-write`** 패턴으로 Plan Mode 흐름을 재현할 수 있습니다
- `approval_policy = "never"` 는 CI/비대화형 환경에 적합합니다
- `--yolo` 플래그는 모든 제한을 해제하므로 신중하게 사용해야 합니다
