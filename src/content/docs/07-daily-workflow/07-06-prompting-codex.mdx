Codex는 단순한 채팅 응답기가 아니라 파일 읽기, 파일 수정, 명령 실행, 도구 호출을 반복하는 코딩 에이전트입니다. 따라서 좋은 프롬프트는 "무엇을 해줘"보다 한 단계 더 구체적이어야 합니다. 공식 문서는 Codex가 검증할 수 있는 작업일수록 결과 품질이 좋아지고, 복잡한 작업은 작고 집중된 단계로 나누는 것이 좋다고 설명합니다.

## 기본 프롬프트 구조

공식 Best practices는 좋은 기본 프롬프트에 네 가지 요소를 넣으라고 권장합니다.

| 요소 | 질문 | 예시 |
|------|------|------|
| Goal | 무엇을 바꾸거나 만들 것인가? | "설정 화면의 저장 버그를 수정해 줘" |
| Context | 어떤 파일, 오류, 문서, 재현 절차가 중요한가? | "`src/settings`와 `tests/settings`를 먼저 봐 줘" |
| Constraints | 지켜야 할 제약은 무엇인가? | "API 응답 형태는 바꾸지 마" |
| Done when | 완료 조건은 무엇인가? | "회귀 테스트가 실패했다가 통과해야 해" |

이 네 가지를 모두 쓰면 Codex가 추측해야 하는 부분이 줄어듭니다. 특히 "Done when"은 Codex가 스스로 검증 루프를 돌 수 있게 만드는 핵심입니다.

## 바로 쓸 수 있는 템플릿

### 기능 구현

```text
Goal:
사용자가 대시보드에서 프로젝트를 즐겨찾기할 수 있게 해 줘.

Context:
- UI는 src/app/dashboard 아래에 있어.
- 기존 상태 관리는 src/lib/projects.ts 패턴을 따라.
- 비슷한 토글 UI는 src/components/FeatureToggle.tsx를 참고해.

Constraints:
- 새 상태 관리 라이브러리는 추가하지 마.
- 기존 API 응답 필드는 바꾸지 마.
- 변경 범위가 커지면 먼저 계획을 제안해.

Done when:
- 즐겨찾기 토글이 새로고침 후에도 유지된다.
- 관련 테스트를 추가하거나 기존 테스트를 갱신한다.
- pnpm test --filter dashboard를 실행하고 결과를 보고한다.
```

### 버그 수정

```text
Bug:
설정 화면에서 "저장됨" 메시지가 뜨지만 새로고침하면 값이 사라진다.

Repro:
1. pnpm dev 실행
2. /settings 접속
3. "알림 받기"를 켠다
4. Save 클릭
5. 페이지 새로고침
6. 토글이 다시 꺼져 있다

Constraints:
- API shape는 바꾸지 마.
- 가능한 가장 작은 회귀 테스트를 추가해.

Done when:
- 먼저 버그를 재현하고 원인을 설명한다.
- 수정 후 같은 절차로 다시 검증한다.
- 실행한 명령과 결과를 마지막에 요약한다.
```

### 코드 리뷰

```text
현재 브랜치의 변경사항을 리뷰해 줘.

Focus:
- 사용자에게 보이는 동작 회귀
- 데이터 손실 가능성
- 테스트 누락
- 접근성 회귀

Output:
- 심각도 순으로 findings를 먼저 써 줘.
- 각 finding에는 파일과 줄 번호를 포함해.
- 문제가 없으면 남은 리스크와 실행하지 못한 검증만 말해 줘.
```

## 큰 작업은 먼저 계획시킨다

작업이 모호하거나 범위가 크면 바로 구현을 맡기지 말고 먼저 계획을 요청합니다. Codex CLI에서는 `/plan` 또는 `Shift`+`Tab`을 사용해 Plan Mode로 전환할 수 있습니다.

```text
/plan
결제 리팩터링을 하려고 한다.
먼저 관련 모듈을 읽고 다음을 정리해 줘.

1. 변경해야 할 파일 후보
2. 위험한 호환성 지점
3. 최소 구현 순서
4. 각 단계의 검증 방법

아직 파일은 수정하지 마.
```

계획을 검토한 뒤 구현을 요청할 때는 계획 전체를 다시 설명하지 말고, 승인한 범위와 바뀐 조건만 알려주는 편이 좋습니다.

```text
좋아. 1단계와 2단계만 진행해.
공개 API 이름은 유지하고, 테스트는 기존 billing 테스트 파일에 추가해.
```

## 컨텍스트를 명시적으로 붙인다

IDE 확장은 열려 있는 파일과 선택 영역을 자동으로 컨텍스트에 포함합니다. CLI에서는 경로를 직접 언급하거나 composer에서 `@` 자동완성, `/mention`을 사용해 파일을 붙이는 습관이 필요합니다.

```text
@src/lib/auth/session.ts @src/lib/auth/session.test.ts
세션 만료 로직을 설명하고, 시간대나 경쟁 조건 때문에 실패할 수 있는 지점을 찾아 줘.
수정하지 말고 검증 아이디어만 제안해.
```

공식 Workflows 문서는 버그 수정처럼 재현 가능한 작업에서는 고수준 설명보다 재현 절차와 제약이 더 중요하다고 안내합니다. "안 돼요"보다 "어떤 명령을 실행하고 어떤 결과가 나오는지"를 주는 편이 낫습니다.

## 완료 기준은 검증 명령으로 쓴다

Codex에게 "잘 고쳐줘"라고만 요청하면 완료 판단이 흐려집니다. 가능한 한 작고 빠른 검증 명령을 지정하세요.

| 상황 | 좋은 완료 기준 |
|------|----------------|
| UI 버그 | "수정 후 같은 재현 절차를 다시 실행하고 결과를 보고" |
| 유닛 로직 | "관련 테스트 파일 하나를 추가하고 해당 테스트만 실행" |
| 타입 변경 | "`pnpm typecheck`와 영향받은 테스트 실행" |
| 문서 수정 | "`pnpm build`로 MDX 빌드 검증" |
| 리뷰 | "`/review`로 uncommitted changes 리뷰 후 findings 반영" |

검증이 오래 걸리거나 환경이 없을 수 있다면 "실행할 수 없으면 이유와 대신 확인한 내용을 보고"라고 조건을 붙입니다.

## 반복 지침은 AGENTS.md로 옮긴다

매번 같은 말을 반복하고 있다면 프롬프트가 아니라 `AGENTS.md`에 들어갈 내용일 가능성이 큽니다. 공식 AGENTS.md 문서는 Codex가 작업 전에 이 파일을 읽고, 글로벌 지침과 프로젝트 지침을 계층적으로 합친다고 설명합니다.

<ConceptCard title="AGENTS.md에 넣기 좋은 내용">

- 프로젝트 실행, 테스트, 린트 명령
- 선호 패키지 매니저와 금지된 의존성
- 코드 스타일과 PR 기대사항
- 보안상 금지해야 하는 작업
- "완료"의 기준과 검증 순서

</ConceptCard>

반대로 한 번만 필요한 버그 재현 절차, 이번 작업의 범위, 임시 제약은 프롬프트에 남겨두는 편이 좋습니다.

## 반복 작업은 Skill로 만든다

공식 Best practices는 반복되는 워크플로를 긴 프롬프트로 계속 수행하지 말고 Skill로 포장하라고 권장합니다. Skill은 `SKILL.md`와 필요한 스크립트, 참고 자료를 묶어 Codex가 특정 작업에 일관되게 적용하도록 만드는 단위입니다.

Skill로 옮기기 좋은 신호는 다음과 같습니다.

- 같은 프롬프트를 세 번 이상 반복해서 쓴다
- 매번 같은 체크리스트를 붙인다
- 특정 로그, 릴리스 노트, PR 리뷰처럼 입력과 출력이 비교적 고정되어 있다
- 팀원이 같은 작업을 같은 방식으로 해야 한다

처음부터 완벽한 Skill을 만들 필요는 없습니다. 대표 사례 하나로 시작하고, 실제 사용 중 반복적으로 고치는 내용을 반영해 개선하는 방식이 더 안전합니다.

## 핵심 정리

- 좋은 Codex 프롬프트는 **Goal, Context, Constraints, Done when** 네 요소를 포함합니다
- 복잡하거나 모호한 작업은 `/plan` 또는 `Shift`+`Tab`으로 먼저 계획하게 하세요
- CLI에서는 파일 경로, `@` mention, `/mention`으로 컨텍스트를 명시적으로 붙이는 습관이 중요합니다
- 완료 기준은 가능한 한 테스트, 재현 절차, 빌드 같은 검증 가능한 형태로 씁니다
- 반복되는 팀 규칙은 `AGENTS.md`, 반복되는 작업 절차는 Skill로 옮기면 프롬프트 품질이 안정됩니다
