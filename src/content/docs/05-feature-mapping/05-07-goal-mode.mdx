:::caution[문서 상태]
2026년 5월 6일 기준 OpenAI의 공개 슬래시 커맨드 문서에는 아직 `/goal` 항목이 빠져 있습니다. 다만 OpenAI 공식 `openai/codex` v0.128.0 릴리스 노트와 관련 PR에는 persisted `/goal` workflow, App Server API, model tools, runtime continuation, TUI control이 추가되었다고 명시되어 있습니다. 이 페이지는 그 공식 릴리스와 v0.128.0 소스 기준으로 설명합니다.
:::

## /goal이 하는 일

`/goal`은 한 번의 프롬프트에 답하고 끝나는 방식이 아니라, **스레드에 지속되는 목표(objective)를 저장하고 Codex가 그 목표를 향해 여러 턴에 걸쳐 계속 작업하도록 하는 기능**입니다.

v0.128.0 릴리스 노트는 이 기능을 다음 구성요소로 설명합니다.

- persisted `/goal` workflows
- App Server goal API
- model-facing goal tools
- runtime continuation
- TUI controls for create, pause, resume, clear

즉 `/goal`은 단순한 프롬프트 매크로가 아닙니다. 목표 상태를 스레드 단위로 저장하고, 현재 목표가 active, paused, budget-limited, complete 중 어떤 상태인지 추적합니다. 관련 PR에는 `thread_goals` state table, 목표별 usage accounting, stale update 보호, 목표 완료 처리 등이 포함되어 있습니다.

<ConceptCard title="/plan과 /goal의 차이">
`/plan`은 구현 전에 계획을 세우는 모드입니다. 반면 `/goal`은 목표를 저장하고, 목표가 완료될 때까지 작업을 이어가게 하는 실행 흐름입니다.

범위가 애매하면 먼저 `/plan`으로 계획을 만들고, 완료 기준이 분명해졌을 때 `/goal`로 전환하는 편이 안전합니다.
</ConceptCard>

## 안정성 수준

공식 Feature Maturity 문서에서 **Under development**는 아직 사용할 준비가 되지 않은 단계로 설명됩니다. v0.128.0 소스의 feature registry에서 `goals`는 `Stage::UnderDevelopment`, `default_enabled: false`로 등록되어 있습니다.

따라서 `/goal`은 기본값으로 꺼져 있으며, 팀 기본 워크플로에 바로 넣기보다는 개인 실험이나 제한된 파일 범위의 검증 가능한 작업에서 먼저 사용해야 합니다.

## 활성화 방법

먼저 버전을 확인합니다.

```bash
codex --version
```

`0.128.0` 이상이어야 합니다. feature flag 상태는 다음 명령으로 확인할 수 있습니다.

```bash
codex features list
```

공식 CLI features 문서에 따르면 feature flag는 `codex features enable <feature>` 또는 `codex features disable <feature>`로 영구 변경할 수 있고, 이 값은 `~/.codex/config.toml`에 저장됩니다.

```bash
codex features enable goals
```

프로필을 사용 중인 세션에서 실행하면 루트 설정이 아니라 해당 profile 설정에 저장됩니다.

직접 설정 파일을 편집한다면 다음처럼 작성합니다.

```toml
# ~/.codex/config.toml

suppress_unstable_features_warning = true

[features]
goals = true
```

:::caution[위치 주의]
`suppress_unstable_features_warning`은 `[features]` 안의 키가 아니라 **최상위 config 키**입니다. TOML에서는 `[features]` 아래에 쓰면 `features.suppress_unstable_features_warning`로 해석될 수 있으므로, 위 예시처럼 `[features]` 섹션보다 먼저 두세요.
:::

설정을 바꾼 뒤 기존 TUI 세션에서 `/goal`이 보이지 않으면 Codex를 재시작하고 다시 확인하세요.

## suppress_unstable_features_warning

`suppress_unstable_features_warning = true`는 under-development feature flag를 켰을 때 표시되는 경고를 숨깁니다.

이 설정이 하는 일은 **경고 메시지를 숨기는 것뿐**입니다.

- `goals` 기능을 활성화하지 않습니다
- under-development 기능을 stable로 만들지 않습니다
- 버그, 동작 변경, 제거 가능성을 낮추지 않습니다

목표 모드를 켠 뒤 매번 경고가 보여 작업 흐름을 방해할 때만 사용하세요. 팀 공유 `.codex/config.toml`에는 이 값을 넣기 전에, 팀원이 under-development 기능을 의도적으로 받아들이는지 먼저 합의하는 편이 좋습니다.

## 기본 사용 흐름

목표를 만들 때는 `/goal` 뒤에 objective를 직접 씁니다.

```text
/goal src/auth의 refresh token race condition을 재현 테스트로 고정하고 수정해줘. 완료 기준: pnpm test auth 통과와 변경 diff 설명.
```

현재 목표 요약을 보고 싶으면 인자 없이 실행합니다.

```text
/goal
```

작업을 잠시 멈추거나 다시 이어가려면 TUI control을 사용합니다. v0.128.0 릴리스 노트는 pause/resume/clear를 언급하고, TUI PR은 `/goal pause`, `/goal unpause`, `/goal clear` 변형을 설명합니다.

```text
/goal pause
/goal unpause
/goal clear
```

이미 목표가 있는 상태에서 새 목표를 입력하면 TUI가 교체 확인 메뉴를 표시합니다. 이는 실수로 진행 중인 목표를 덮어쓰는 일을 줄이기 위한 동작입니다.

## 모델 도구 관점

v0.128.0에는 모델이 사용할 수 있는 목표 도구도 추가되었습니다.

- `get_goal`: 현재 목표와 상태를 확인합니다
- `create_goal`: 사용자가 명시적으로 요청한 경우에만 목표를 만듭니다
- `update_goal`: 목표를 완료 상태로 표시합니다

공식 PR 설명상 모델은 일반 작업 요청을 마음대로 goal로 추론하면 안 됩니다. 또한 `update_goal`은 완료 처리만 노출하고, pause/resume/clear는 사용자나 런타임이 제어합니다. 이 제한은 목표 상태가 사용자 의도와 다르게 바뀌는 것을 막기 위한 설계입니다.

## 좋은 목표 작성법

`/goal`에는 넓은 방향보다 검증 가능한 완료 기준을 넣는 것이 중요합니다.

좋은 예:

```text
/goal 결제 실패 시 재시도 로직을 구현해줘. 완료 기준: 실패 케이스 단위 테스트 2개 추가, 기존 결제 테스트 통과, 변경 파일과 남은 리스크 요약.
```

피해야 할 예:

```text
/goal 앱을 전반적으로 개선해줘
```

목표에는 최소한 다음 정보를 포함하세요.

- 바꾸려는 영역 또는 파일 범위
- 성공 조건
- 실행할 검증 명령
- 멈춰야 할 조건
- 사용자 확인이 필요한 위험한 작업

## 사용하지 않는 편이 나은 경우

다음 상황에서는 `/goal`보다 `/plan`이나 일반 프롬프트가 낫습니다.

- 요구사항이 아직 모호한 경우
- 대규모 리팩터링인데 변경 범위가 정해지지 않은 경우
- 프로덕션 데이터, 인증, 결제처럼 수동 검토가 필요한 영역
- 테스트나 수동 확인 절차가 없는 작업
- 팀원이 under-development 기능 사용에 동의하지 않은 공유 저장소

## 문제 해결

`/goal`이 보이지 않으면 다음 순서로 확인합니다.

1. `codex --version`이 `0.128.0` 이상인지 확인합니다.
2. `codex features list`에서 `goals`가 `true`인지 확인합니다.
3. `~/.codex/config.toml`에서 `[features] goals = true`가 올바르게 들어갔는지 확인합니다.
4. `suppress_unstable_features_warning`을 `[features]` 아래에 잘못 넣지 않았는지 확인합니다.
5. Codex TUI를 완전히 재시작합니다.

그래도 보이지 않으면 현재 설치된 빌드나 사용 중인 표면에서 `/goal` TUI가 아직 노출되지 않은 상태일 수 있습니다. 이 경우 같은 목표를 일반 프롬프트나 `/plan`으로 먼저 처리하고, 업데이트 후 다시 확인하세요.

## 핵심 정리

- `/goal`은 v0.128.0에서 추가된 persisted goal workflow입니다
- `goals` feature flag는 under-development이며 기본값은 `false`입니다
- 활성화는 `codex features enable goals` 또는 `[features] goals = true`로 합니다
- `suppress_unstable_features_warning = true`는 경고만 숨기며 기능 안정성을 바꾸지 않습니다
- 목표에는 파일 범위, 완료 기준, 검증 명령, 중단 조건을 함께 적어야 합니다
