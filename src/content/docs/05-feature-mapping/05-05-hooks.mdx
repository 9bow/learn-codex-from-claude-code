## Claude Code의 Hooks 체계

Claude Code는 `~/.claude/settings.json` 또는 `.claude/settings.json` 의 `hooks` 키에 훅을 정의합니다. 지원 이벤트는 다음과 같습니다.

| 이벤트 | 발생 시점 |
|--------|-----------|
| `PreToolUse` | 도구 실행 직전 |
| `PostToolUse` | 도구 실행 완료 후 |
| `PostToolUseFailure` | 도구 실행 실패 후 |
| `Stop` | 세션 종료 시 |
| `SessionStart` | 세션 시작 시 |
| `SessionEnd` | 세션 종료 시 |
| `UserPromptSubmit` | 사용자 프롬프트 제출 시 |
| `PermissionRequest` | 권한 요청 시 |

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "python3 .claude/hooks/validate.py"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "prettier --write $TOOL_OUTPUT_FILE"
          }
        ]
      }
    ]
  }
}
```

## Codex의 Hooks 체계

:::caution[기능 플래그 필요]
OpenAI 공식 Hooks 문서는 Codex hooks가 `config.toml`의 `features.codex_hooks = true` 플래그 뒤에 있다고 설명합니다. hooks를 이전할 때는 먼저 이 플래그를 켜고, 이벤트·스키마는 공식 Hooks 문서에서 확인하세요.
:::

Hooks는 `~/.codex/hooks.json` 파일이나 `config.toml` 의 `[hooks]` 인라인 섹션으로 정의합니다.

```toml
# ~/.codex/config.toml
[features]
codex_hooks = true

[[hooks.PreToolUse]]
matcher = "^Bash$"

[[hooks.PreToolUse.hooks]]
type = "command"
command = "/usr/bin/python3 .codex/hooks/policy.py"
timeout = 30
statusMessage = "Bash 명령 정책 검사 중"
```

또는 별도 `~/.codex/hooks.json` 파일로 분리할 수 있습니다.

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "^Bash$",
        "hooks": [
          {
            "type": "command",
            "command": "/usr/bin/python3 .codex/hooks/policy.py",
            "timeout": 30,
            "statusMessage": "Bash 명령 정책 검사 중"
          }
        ]
      }
    ]
  }
}
```

## 이벤트 차이

Claude Code와 Codex 모두 여러 lifecycle event를 지원하지만, 이벤트 이름과 세부 입출력 스키마가 완전히 같지는 않습니다.

| 이벤트 | Claude Code | Codex |
|--------|-------------|-------|
| `PreToolUse` | 지원 | 지원 |
| `PostToolUse` | 지원 | 지원 |
| `PostToolUseFailure` | 지원 | 미지원 |
| `SessionStart` | 지원 | 지원 |
| `SessionEnd` | 지원 | 직접 대응 없음 |
| `Stop` | 지원 | 지원 |
| `UserPromptSubmit` | 지원 | 지원 |
| `PermissionRequest` | 지원 | 지원 |

:::note[참고]
Codex hooks의 matcher, 입력 JSON, 출력 JSON은 이벤트별로 다릅니다. 특히 `PreToolUse`는 일부 shell 호출을 아직 모두 가로채지 못하므로, 보안 경계로만 의존하지 말고 sandbox와 approval policy를 함께 사용하세요.
:::

## 활성화 방법

`config.toml`에 feature flag를 추가한 뒤 `hooks.json` 또는 인라인 `[hooks]` 설정을 작성합니다.

```toml
[features]
codex_hooks = true
```

## 프로젝트 신뢰(trust) 요건

Codex는 프로젝트 `.codex/` 폴더의 hooks 설정이 **신뢰된(trusted) 프로젝트** 에서만 로드됩니다. 처음 `codex` 를 실행할 때 프로젝트 신뢰 여부를 확인하는 프롬프트가 나타납니다.

<ConceptCard title="변환 시 고려사항">
Claude Code hooks에서 Codex로 이전할 때 주의할 점이 있습니다.

1. **기능 플래그 확인**: `features.codex_hooks = true` 를 먼저 설정하세요.
2. **matcher 정규식**: Codex는 `matcher` 에 정규식(`^Bash$`)을 사용합니다. Claude Code의 단순 문자열 매처와 다를 수 있습니다.
3. **timeout 필드**: Codex hooks에는 `timeout` 필드를 명시하는 것이 권장됩니다.
4. **환경 변수**: 훅 스크립트에서 사용하는 환경 변수 이름이 다를 수 있으므로 확인이 필요합니다.
</ConceptCard>

## 핵심 정리

- Codex hooks는 `features.codex_hooks = true` 플래그를 켜야 사용할 수 있습니다
- 훅 정의는 `config.toml` 의 `[hooks]` 섹션 또는 별도 `hooks.json` 파일로 작성합니다
- Codex는 `PreToolUse`, `PermissionRequest`, `PostToolUse`, `SessionStart`, `UserPromptSubmit`, `Stop` 등을 지원합니다
- `matcher` 는 정규식 형식으로 작성하며, `timeout` 필드를 함께 명시하는 것이 권장됩니다
- 프로젝트 `.codex/` 의 훅은 신뢰된 프로젝트에서만 로드되므로, 신뢰 설정을 먼저 확인하세요
