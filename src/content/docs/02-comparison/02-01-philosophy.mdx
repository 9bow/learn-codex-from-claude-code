## 두 도구의 근본적 차이

Claude Code와 Codex는 "AI가 코드를 수정할 때 얼마나 자유롭게 행동해야 하는가"라는 질문에 서로 다른 철학으로 답합니다.

| 도구 | 기본 철학 | 한 줄 요약 |
|------|-----------|------------|
| **Claude Code** | Permission-first | "허용된 것만 한다" |
| **Codex** | Sandbox + Approval | "격리된 환경에서 하고, 필요할 때만 묻는다" |

## Claude Code: Permission-first

Claude Code는 작업을 실행하기 전에 **권한 규칙** 을 먼저 확인합니다. 기본 모드(`default`)에서는 파일 쓰기나 명령어 실행마다 사용자 승인을 요청합니다.

```json
// ~/.claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Read(./src/**)"
    ],
    "deny": [
      "Bash(curl *)"
    ]
  }
}
```

권한 모드:

| 모드 | 동작 |
|------|------|
| `default` | 허용되지 않은 모든 작업에 승인 요청 |
| `acceptEdits` | 파일 편집은 자동, 명령어는 승인 |
| `plan` | 실행 없이 계획만 수립 |
| `auto` | 모든 작업 자동 실행 |
| `bypassPermissions` | 모든 권한 검사 우회 |

<ConceptCard title="Permission-first의 강점과 약점">
**강점**: 세밀한 제어. 허용·거부 규칙을 glob 패턴으로 정확히 지정 가능. 어떤 작업이 실행될지 완전히 예측 가능.

**약점**: 초기 설정 비용. 새 프로젝트마다 권한 규칙을 쌓아야 함. 자동화 환경에서는 `auto` 모드로 전환해야 하는 번거로움.
</ConceptCard>

## Codex: Sandbox + Approval

Codex는 작업 자체를 **격리된 환경(sandbox)** 에서 실행하고, 범위를 벗어나는 작업만 승인을 요청합니다. "허용 목록을 만드는" 대신 "경계를 설정하는" 접근입니다.

<Compare>
  <Fragment slot="left">
```json
// Claude Code 권한 설정 예시
{
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(npm *)",
      "Write(./src/**)",
      "Read(**)"
    ],
    "deny": ["Bash(rm -rf *)"]
  }
}
```

  </Fragment>
  <Fragment slot="right">
```toml
# Codex sandbox 설정 예시
approval_policy = "on-request"
sandbox_mode = "workspace-write"
# → 작업 디렉토리 내 쓰기 허용
# → 범위 밖 작업시 자동 승인 요청
```

  </Fragment>
</Compare>

### Sandbox 모드

| 모드 | 허용 범위 |
|------|-----------|
| `read-only` | 파일 읽기만 허용. 쓰기/실행 모두 승인 필요 |
| `workspace-write` | 작업 디렉토리 내 읽기·쓰기 허용 (기본값) |
| `danger-full-access` | 모든 파일·네트워크 접근 허용 |

### Approval Policy

| 정책 | 동작 |
|------|------|
| `untrusted` | 모든 작업 승인 필요 |
| `on-request` | 에이전트가 필요할 때만 요청 (기본값) |
| `never` | 승인 없이 자동 실행 |
| `granular` | 세분화된 규칙 기반 |

:::note[참고]
이전 버전의 `on-failure` 정책은 deprecated되었으며 `on-request` 로 대체되었습니다.
:::

## 두 철학의 실질적 차이

<Compare>
  <Fragment slot="left">
**새 프로젝트 시작 시**:
- 첫 실행에서 권한 요청이 많이 발생
- 허용 규칙을 하나씩 추가하며 튜닝
- 이후 안정적이고 예측 가능한 동작

**CI/CD 파이프라인**:
- `auto` 모드 또는 상세 허용 규칙 필요
- 규칙을 `.claude/settings.json` 에 커밋
</Fragment>
<Fragment slot="right">
**새 프로젝트 시작 시**:
- sandbox가 기본 적용되므로 바로 실행 가능
- `workspace-write` + `on-request` 로 대부분 커버
- 별도 허용 목록 없이도 시작 가능

**CI/CD 파이프라인**:
- `approval_policy = "never"` 로 전환
- `sandbox_mode = "danger-full-access"` 검토
- `codex exec --ask-for-approval never "..."` 플래그
</Fragment>
</Compare>

## 철학 차이가 Skills와 Hooks에도 반영된다

이 철학 차이는 도구 전체에 일관되게 반영됩니다.

- **Claude Code hooks**: `PreToolUse` 에서 권한을 차단하거나 수정하는 방식
- **Codex hooks**: `features.codex_hooks = true` 가 필요한 **실험적 기능** 이며, sandbox 안에서 추가 정책 레이어로 동작

- **Claude Code subagents**: 부모의 권한 범위를 상속
- **Codex subagents**: 각 에이전트가 독립적인 sandbox를 가질 수 있음

## 핵심 정리

- Claude Code는 **허용 목록(allowlist) 기반** 의 permission-first 모델입니다
- Codex는 **격리(sandbox) 기반** 의 접근으로, 경계 안에서는 자유롭게 동작합니다
- Codex 기본값은 `approval_policy = "on-request"` + `sandbox_mode = "workspace-write"` 입니다
- Claude Code의 세밀한 glob 규칙은 Codex에서 sandbox 경계 + granular 정책으로 대체됩니다
- CI/CD에서는 Codex `approval_policy = "never"` 가 Claude Code `auto` 모드에 대응합니다
- 두 철학은 우열이 없으며, 팀·프로젝트 성격에 따라 선택하면 됩니다
